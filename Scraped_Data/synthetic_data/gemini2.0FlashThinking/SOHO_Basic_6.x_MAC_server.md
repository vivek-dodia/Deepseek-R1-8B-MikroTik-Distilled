## MikroTik RouterOS MAC Server Documentation (v6.x - Basic SOHO)

**Topic:** MAC Server
**RouterOS Version:** 6.x
**Network Scale:** SOHO (Small Office/Home Office)
**Complexity Level:** Basic

**Introduction:**

This document provides a comprehensive guide to configuring and managing the MAC Server feature on MikroTik RouterOS v6.x in a basic SOHO environment.  MAC Server in RouterOS allows the router to act as a central point for managing MAC addresses, often used in conjunction with other services like DHCP Server to control network access at the data link layer. This document will cover configuration via both CLI and REST API, debugging, security, and SOHO-specific considerations.

**2. CLI Configuration with Inline Comments:**

```routeros
# --- MAC Server Configuration ---

# 1. Enable MAC Server on the LAN interface (e.g., ether2 - adjust to your setup)
/interface mac-server
add interface=ether2 # Specify the LAN interface where MAC Server will be active
set 0 enabled=yes   # Enable the MAC Server instance (0 is usually the first instance created)

# 2. Configure Allowed MAC Address List (Optional but common for access control)
/interface mac-server access-list
add mac-address=00:11:22:33:44:55 comment="Desktop PC - Allowed" # Add allowed MAC address with a comment
add mac-address=AA:BB:CC:DD:EE:FF comment="Laptop - Allowed"  # Add another allowed MAC address

# 3. Configure action for non-allowed MAC addresses (Optional - default is 'drop')
/interface mac-server settings
set non-allowed-action=drop # Options: drop, forward, accept. 'drop' is typical for access control.

# 4. Verify MAC Server status
/interface mac-server print
# Check if 'enabled' is yes and 'interfaces' shows the correct interface.

/interface mac-server access-list print
# Verify the list of allowed MAC addresses.

/interface mac-server settings print
# Verify the 'non-allowed-action' setting.

# --- DHCP Server Configuration (Commonly used with MAC Server in SOHO - Assuming DHCP is also desired) ---

# 5. Ensure DHCP Server is running on the same LAN interface (if you want to assign IPs to allowed MACs)
/ip dhcp-server
print # Check if DHCP Server is already configured and running on ether2 or your LAN interface.

# If DHCP Server is not configured, you can configure it (Basic example - adjust to your network)
# /ip dhcp-server
# add address-pool=dhcp_pool disabled=no interface=ether2 lease-time=10m name=dhcp1
# /ip pool
# add name=dhcp_pool ranges=10.0.0.10-10.0.0.254
# /ip network
# add address=10.0.0.0/24 gateway=10.0.0.1 dhcp-server=dhcp1

# Note: This DHCP configuration is a basic example and may need adjustments for your specific network needs.
```

**3. REST API Implementation (Python Code):**

```python
import requests
import json

ROUTER_IP = "192.168.88.1" # Replace with your Router IP
API_USER = "api_user"      # Replace with your API username
API_PASSWORD = "api_password" # Replace with your API password
INTERFACE_NAME = "ether2" # Replace with your LAN interface name
ALLOWED_MACS = [
    {"mac_address": "00:11:22:33:44:55", "comment": "Desktop PC - API Allowed"},
    {"mac_address": "AA:BB:CC:DD:EE:FF", "comment": "Laptop - API Allowed"}
]

BASE_URL = f"http://{ROUTER_IP}/rest"
HEADERS = {'Content-Type': 'application/json'}

def api_request(method, endpoint, data=None):
    url = f"{BASE_URL}{endpoint}"
    try:
        response = requests.request(method, url, headers=HEADERS, json=data, auth=(API_USER, API_PASSWORD), verify=False) # verify=False for simplicity, consider certificate verification in production
        response.raise_for_status() # Raise HTTPError for bad responses (4xx or 5xx)
        return response.json()
    except requests.exceptions.HTTPError as http_err:
        print(f"HTTP error occurred: {http_err}")
        if response.text:
            print(f"Response content: {response.text}") # Print response body for more details
        return None
    except requests.exceptions.RequestException as req_err:
        print(f"Request error occurred: {req_err}")
        return None

def enable_mac_server():
    endpoint = "/interface/mac-server"
    data = {"interface": INTERFACE_NAME, "enabled": True}
    response = api_request("POST", endpoint, data) # Using POST to create a new MAC Server instance if none exists
    if response:
        print("MAC Server enabled successfully.")
    else:
        print("Failed to enable MAC Server.")

def add_allowed_mac_address(mac_address, comment):
    endpoint = "/interface/mac-server/access-list"
    data = {"mac-address": mac_address, "comment": comment}
    response = api_request("POST", endpoint, data)
    if response:
        print(f"MAC Address {mac_address} added to allowed list.")
    else:
        print(f"Failed to add MAC Address {mac_address} to allowed list.")

def set_non_allowed_action(action="drop"):
    endpoint = "/interface/mac-server/settings"
    data = {"non-allowed-action": action}
    response = api_request("PATCH", endpoint, data) # Using PATCH to modify existing settings
    if response:
        print(f"Non-allowed action set to: {action}")
    else:
        print(f"Failed to set non-allowed action.")


if __name__ == "__main__":
    enable_mac_server()
    for mac_entry in ALLOWED_MACS:
        add_allowed_mac_address(mac_entry["mac_address"], mac_entry["comment"])
    set_non_allowed_action()
```

**Python Script Explanation:**

* **`requests` library:**  Used for making HTTP requests to the RouterOS REST API.
* **`api_request` function:** Handles generic API requests (GET, POST, PATCH, DELETE) with error handling.
* **`enable_mac_server` function:** Enables MAC Server on the specified interface.
* **`add_allowed_mac_address` function:** Adds a MAC address to the allowed access list.
* **`set_non_allowed_action` function:** Configures the action for non-allowed MAC addresses.
* **Error Handling:** Includes `try-except` blocks to catch HTTP errors and request exceptions, printing error messages.
* **`verify=False`:**  Used for simplicity in this example. **In a production environment, you should implement proper certificate verification for secure API communication.**

**To use this script:**

1. **Enable REST API on your MikroTik RouterOS:**  `/ip service enable api-ssl` (or `/ip service enable api` for non-SSL, not recommended for sensitive data).
2. **Create an API user:** `/user add name=api_user password=api_password group=full` (adjust permissions as needed).
3. **Install `requests` library:** `pip install requests`.
4. **Replace placeholder values:** `ROUTER_IP`, `API_USER`, `API_PASSWORD`, `INTERFACE_NAME` with your actual values.
5. **Run the script:** `python your_script_name.py`.

**5. Version-Specific Considerations (RouterOS v6.x):**

* **Basic Functionality Stable:** MAC Server functionality in v6.x is generally stable for basic SOHO setups.
* **Feature Set:**  v6.x MAC Server has the core features needed for basic access control: enabling/disabling, access list, non-allowed action.  More advanced features introduced in later versions (like MAC Authentication protocol options, RADIUS integration for MAC Server - less relevant for basic SOHO) are not present in v6.x.
* **API Compatibility:** The REST API in v6.x may have slight differences in structure compared to v7.  The examples provided are generally compatible with v6.x REST API for MAC Server.  Always refer to MikroTik documentation for exact API paths if encountering issues.
* **Security Updates:** RouterOS v6.x is in legacy support. Ensure you are running the latest v6.x version available for security patches, although long-term security updates may be limited.  Consider upgrading to v7 for ongoing security support if feasible for your hardware and network needs.

**6. Security Hardening Measures:**

* **MAC Address Filtering as Supplementary Security:**  MAC address filtering is **not a primary security mechanism**. MAC addresses can be spoofed.  Use it as a **supplementary layer** for basic access control in SOHO, not as the sole security measure.
* **Strong RouterOS Passwords:**  Use strong and unique passwords for all RouterOS user accounts, especially the `admin` account and any API users.
* **Limit API Access:** If using the API, restrict API user permissions to only what is necessary (avoid `full` group if possible). Use HTTPS (API-SSL) for API communication to encrypt credentials in transit. Consider firewall rules to limit API access to specific IP ranges if needed.
* **Disable Unnecessary Services:** Disable any RouterOS services that are not required for your SOHO setup to reduce the attack surface.
* **Keep RouterOS Updated (Within v6.x):**  Apply the latest RouterOS v6.x updates provided by MikroTik for bug fixes and security patches.
* **Firewall Rules:** Implement robust firewall rules to protect your SOHO network from unauthorized external access, regardless of MAC Server configuration.

**7. Performance Optimization Tips:**

* **MAC Server Performance in SOHO:** For basic SOHO networks with a typical number of devices, the performance impact of MAC Server is **negligible**.  MAC Server operations are lightweight and do not consume significant CPU or memory resources in most SOHO scenarios.
* **Access List Size:** A reasonably sized MAC address access list (dozens or even hundreds of entries in a SOHO) will not significantly impact performance.  Performance impact might become noticeable only with extremely large access lists (thousands of entries), which is unlikely in a typical SOHO.
* **Efficient Rule Processing:** RouterOS is generally efficient in processing MAC Server rules. No specific performance optimization is usually required for basic SOHO MAC Server usage.
* **Hardware Considerations:** Ensure your MikroTik router hardware is adequately sized for your overall SOHO network needs (routing, firewall, NAT, etc.). If the router is already handling heavy loads, consider upgrading to a more powerful model if you anticipate significant network growth.

**Real-World Deployment Examples:**

* **Home Network - Parental Control:** Use MAC Server to allow only family devices (laptops, phones, tablets) to connect to the home Wi-Fi network, preventing unauthorized devices from accessing the internet.
* **Small Office - Basic Device Management:** In a small office with a limited number of employees and devices, use MAC Server to control which employee devices are allowed on the office network, providing a simple form of access control.
* **Guest Network Isolation (Simplified):** Although VLANs and dedicated guest Wi-Fi setups are more robust, in a very basic setup, MAC Server could be used on a specific interface to create a rudimentary "guest" network by only allowing pre-approved MAC addresses to connect.  **However, this is not a recommended or secure approach for guest networks. Use proper VLANs and guest Wi-Fi features for secure guest access.**

**Scalability Considerations (SOHO Context):**

* **Limited Scalability Need in SOHO:**  SOHO environments typically have a relatively small and stable number of devices. Scalability of MAC Server itself is usually not a primary concern in SOHO.
* **Router Hardware Limits:** The primary scalability limitation in a SOHO setup is usually the overall capacity of the MikroTik router hardware itself (CPU, memory, interface count, throughput). If your SOHO network grows significantly (beyond typical SOHO scale), you might need to consider a more powerful MikroTik router model.
* **Access List Management:**  Maintaining a large MAC address access list manually can become cumbersome. For larger scale or more dynamic environments, consider using more robust authentication and authorization mechanisms like RADIUS or MikroTik User Manager, which are beyond the scope of basic SOHO MAC Server documentation.

**Monitoring Configurations:**

* **Basic Monitoring:**
    * **Interface Status:** Monitor the interface where MAC Server is enabled to ensure it's up and running. `/interface print`
    * **Logs:** Check RouterOS logs for any MAC Server related events or errors. `/log print follow-name=mac-server`
    * **DHCP Leases (if applicable):** Monitor DHCP leases to see which devices are getting IP addresses and verify if allowed MAC addresses are receiving leases as expected. `/ip dhcp-server lease print`

* **No Dedicated MAC Server Monitoring Tools in Basic RouterOS:**  RouterOS does not have dedicated built-in tools specifically for monitoring MAC Server activity beyond logs and basic interface status. For more advanced monitoring, you might need to integrate with external monitoring systems (SNMP, The Dude, etc.), but this is generally not required for basic SOHO MAC Server usage.

# Universal API Helper Function
def mikrotik_api_call(
    method: str,
    endpoint: str,
    data: dict = None,
    timeout: int = 10
) -> dict:
    '''
    Universal MikroTik API handler with error checking
    '''
    try:
        response = requests.request(
            method,
            f"https://{ROUTER_IP}/rest{endpoint}",
            auth=HTTPBasicAuth(API_USER, API_PASS),
            json=data,
            verify=SSL_VERIFY,
            timeout=timeout
        )
        response.raise_for_status()
        return response.json()
    except requests.exceptions.HTTPError as e:
        print(f"API Error: {e.response.status_code} - {e.response.text}")
        return {"error": str(e)}
